# 架构设计

## 1. 设计目标

- **产品**：类 SillyTavern 的现代化应用（中兼容），优先 MVP 可交付与可演进。
- **体验**：对话、角色、Prompt、知识库/RAG、SSE 流式输出；支持分支回溯。
- **工程**：对外只需一条启动路径——运行单个 `start.bat` / `start.sh` 即可拉起 Node 侧全部必需进程；首次运行可自动 `npm install`。
- **数据**：以**可同步的本地目录 + JSON 文件**为主（对话按 chunk 切分、多级 `index.json`）；办公室与家庭通过 **Syncthing** 同步同一数据目录，且**约定单写者、不同时写入**。
- **扩展**：各能力通过 **API 配置中心**（稳定 `id` / `name`）被引用；对话、RAG、RERANK、Summary 及后续插件可分别绑定不同配置。

## 2. 总体架构

```text
+------------------+       HTTP / SSE +-------------------+
|  Vue3            |  <------------------->  |  Fastify (Node)   |
|  Pinia + Vuetify |       JWT + SSE           |  REST + 流式对话 |
+------------------+                           +---------+---------+
                                                         |
                                                         v +-------------------+
                                              |  本地数据目录      |
                                              |  chat/ config/ kb/ |
                                              |  （Syncthing 同步）|
                                              +-------------------+
```

- **前端**：Vue 3、Pinia、Vuetify；对话壳布局、富文本与思维链分流见 `DOC/03-实现细节.md` §11。
- **后端**：Node.js、Fastify；认证 **JWT**；模型输出 **SSE**。
- **模型协议**：MVP 为 **OpenAI 兼容**（可配置 `baseUrl`、模型名、密钥由服务端保管）。

## 3. 逻辑分层

| 层 | 职责 |
|----|------|
| 表现层 | 页面路由、会话列表、消息树 UI、配置与知识库管理界面 |
| 应用层 | 用例编排：发消息、选分支、绑定 API、触发 RAG / Summary |
| 领域层 | 对话树、chunk 边界、Prompt 合成规则、API 绑定解析规则 |
| 基础设施 | 文件读写、OpenAI 兼容 HTTP 客户端、日志与可观测性 |

## 4. API 配置中心（核心）

- **资源**：每条 API 配置有稳定 **`id`** 与 **`name`**（用户内唯一），含 `baseUrl`、密钥（密文）、默认模型、`capabilities`（如 chat / embedding / rerank）等；字段细则见 `DOC/03-实现细节.md`。
- **绑定**：业务模块不直接散落密钥；通过 **`apiConfigId`** 解析连接参数。**全局默认 → 对话 `apiPreset` → 按 `pluginId` 的 `apiPreset.plugins`**（插件可选用不同预设）；见 `DOC/02-需求说明.md` §3 与 `DOC/03-实现细节.md` §1.2–§1.5。
- **覆盖范围**（MVP 与演进）：
  - 对话主模型（chat）
  - RAG / 向量化相关（embedding、rag_generate 等，按实现拆分）
  - 重排序（rerank）
  - 摘要（summary）
  - 插件实例（V2，每插件可独立配置）

## 5. 数据与存储架构

### 5.1 原则

- **对话内容**：目录 + **chunk 文件**（如每 50～100 轮一切分）；chunk 内 **`meta.links`** 承载 `previous` / `next` / `branches`；**会话级元信息、对话级 API preset、备份参数（`N`/`K`）** 由各级 **`index.json`** 维护（`chat.index.json`、会话根、分支子树）。结构约定见 `DOC/03-实现细节.md` §6–§7。
- **角色、Prompt、知识库、API 配置**：与对话并列子目录（如 `config/`、`kb/`），以 **JSON 或小文件集合** 为主，便于备份与 Syncthing。
- **数据库**：默认**不依赖**独立数据库服务（如 MongoDB），以保证「单脚本启动 + 目录可整体同步」。若未来引入 DB，须通过**仓储抽象**接入，且不破坏数据目录作为部署与同步单元的模型。

### 5.2 Syncthing

- **假设**：办公室与家庭无缝衔接，**同一时刻仅一处写入**该数据目录。
- **实践**：换机前尽量正常退出应用；偶发冲突按 Syncthing 冲突副本规则处理。

### 5.3 标识与顺序

- **`turnId`**：全局唯一、稳定引用（可为哈希，规则需版本化）。
- **`turnOrdinal`**：当前选定路径上的「第 N 轮」，与 `turnId` 分离；分支场景下「第 N 轮」仅对选定路径有意义。

## 6. 关键运行时链路（摘要）

1. 用户登录，颁发 JWT。
2. 打开会话：读取对应 `index.json` 与 chunk 链，按当前分支渲染。
3. 发送消息：解析 **chat** 绑定 → OpenAI 兼容请求 → **SSE** 推送到前端。
4. 若启用 RAG：解析 **embedding / rag / rerank** 绑定 → 检索与重排 → 注入上下文。
5. 可选 Summary：解析 **summary** 有效 preset（对话覆盖全局）→ 写回位置见 `DOC/03-实现细节.md` §10（建议）。
6. 可观测性：请求日志、错误、耗时、token、RAG 命中明细（与 `DOC/02-需求说明.md` 一致）。

## 7. 插件系统（V2 预留）

- 用户侧 preset 与 **包内 API 设定（可含密钥）** 并存：包内设定**仅作用于该插件**，见 `DOC/03-实现细节.md` §1.3。
- 插件**不得**读写其它 API 配置；出站调用**一律经后端转发**。
- MVP 不实现完整插件宿主；架构预留按 `pluginId` 加载与配置隔离的扩展点。

## 8. 安全与多用户（内网）

- **认证**：JWT；API Key **不落前端明文**。
- **隔离**：数据路径按 **`userId`**（或等价租户键）分区，避免跨用户读取。
- **权限**：MVP 不做复杂 RBAC；依赖内网边界与应用内用户隔离。
- **插件**：包内密钥仅服务本插件；**禁止**插件访问他人 API 配置；**禁止**前端直连插件目标 URL（见 `DOC/03-实现细节.md` §1.3、§4）。
