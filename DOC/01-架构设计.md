# 架构设计

## 0. 项目阶段

- **2026-05 起**：**已脱离 MVP**。功能与数据模型按完整产品规划实现；`DOC/02` §2 仅作历史 MVP 对照。
- 文档中仍可能出现的「MVP」多指**当时的技术选型**（如 OpenAI 兼容协议、目录型存储、暂不引入独立 DB），**不等于**当前迭代仍拒绝实现世界书、RAG、JWT 等能力。

## 1. 设计目标

- **产品**：类 SillyTavern 的现代化应用（中兼容），**已脱离 MVP 阶段**（见 `DOC/02` §1.1），按完整产品能力持续演进。
- **体验**：对话、角色、Prompt、知识库/RAG、SSE 流式输出；支持分支回溯。
- **工程**：对外只需一条启动路径——运行单个 `!_start.bat` / `!_start.sh`（`npm start` → 单端口生产）即可拉起应用；**首次运行与 `git pull` 后**可自动 `npm install`（`scripts/ensure-deps.mjs` 按 lock/manifest 指纹）与构建。改代码时用 **`npm run dev`**（双端口 + 热更新，同样会先 `ensure-deps`）。详见 **§9** 与用户 **`README.md`**。
- **数据**：以**可同步的本地目录 + JSON 文件**为主（对话按 chunk 切分、多级 `index.json`）；办公室与家庭通过 **Syncthing** 同步同一数据目录，且**约定单写者、不同时写入**。
- **扩展**：各能力通过 **API 配置中心**（稳定 `id` / `name`）被引用；对话、RAG、RERANK、Summary 及后续插件可分别绑定不同配置。

## 2. 总体架构

```text
+------------------+       HTTP / SSE +-------------------+
|  Vue3            |  <------------------->  |  Fastify (Node)   |
|  Pinia + Vuetify |       JWT + SSE           |  REST + 流式对话 |
+------------------+                           +---------+---------+
                                                         |
                                                         v +-------------------+
                                              |  本地数据目录      |
                                              |  {userId}/chats prompts/ characters/ … |
                                              |  （Syncthing 同步）|
                                              +-------------------+
```

- **前端**：Vue 3、Pinia、Vuetify；对话壳布局、富文本与思维链分流见 `DOC/03-实现细节.md` §11。
- **后端**：Node.js、Fastify；认证 **JWT**；模型输出 **SSE**。
- **模型协议**：当前以 **OpenAI 兼容** 为主（可配置 `baseUrl`、模型名、密钥由服务端保管）；其它供应商经适配层扩展。

## 3. 逻辑分层

| 层 | 职责 |
|----|------|
| 表现层 | 页面路由、会话列表、消息树 UI、配置与知识库管理界面 |
| 应用层 | 用例编排：发消息、选分支、绑定 API、触发 RAG / 插件摘要（Historian · `plot-summary`） |
| 领域层 | 对话树、chunk 边界、Prompt 合成规则、API 绑定解析规则 |
| 基础设施 | 文件读写、OpenAI 兼容 HTTP 客户端、日志与可观测性 |

## 4. API 配置中心（核心）

- **资源**：每条 API 配置有稳定 **`id`** 与 **`name`**（用户内唯一），含 `baseUrl`、密钥（密文）、默认模型、`capabilities`（如 chat / embedding / rerank）等；字段细则见 `DOC/03-实现细节.md`。
- **绑定**：业务模块不直接散落密钥；通过 **`apiConfigId`** 解析连接参数。**全局默认 → 对话功能 `apiPreset` → 插件 `pluginSettings` / 全局插件 settings**（插件可选用不同预设）；见 `DOC/02-需求说明.md` §3 与 `DOC/03-实现细节.md` §1.2–§1.5。
- **覆盖范围**（按能力键绑定，持续扩展）：
  - 对话主模型（chat）
  - RAG / 向量化相关（embedding、rag_generate 等，按实现拆分）
  - 重排序（rerank）
  - 插件实例（每插件可独立配置；**Historian**（`plot-summary`）承担对话摘要，无独立 `summary` 键）

## 5. 数据与存储架构

### 5.1 原则

- **对话内容**：目录 + **chunk 文件**（如每 50～100 轮一切分）；chunk 内 **`meta.links`** 承载 `previous` / `next` / `branches`；**会话级元信息、对话级 API preset、备份参数（`N`/`K`）** 由各级 **`index.json`** 维护（`chat.index.json`、会话根、分支子树）。结构约定见 `DOC/03-实现细节.md` §6–§7。
- **角色卡、Prompt、知识库、API 配置**：与对话并列，按用户存放在 **`data/{userId}/`**（`userId` 为 8 位 hex；种子账号 `00000000`）。认证后由 JWT 解析当前用户，不再使用 `X-User-Id`。例如 `prompts/index.json` + `preset-*.json`、`api-settings.json`、**`characters/{id}.png`** 与 `characters/index.json`、`lorebooks/`。详见 `data/README.md`；角色库实现见 `DOC/03-实现细节.md` **§12**。
- **数据库**：默认**不依赖**独立数据库服务（如 MongoDB），以保证「单脚本启动 + 目录可整体同步」。**对话记忆**使用嵌入式 **LanceDB**（`data/{userId}/memory/` 等，派生索引、可重建）；须经**仓储抽象**接入，且不替代 chunk JSON 权威源。若未来引入其它 DB，同样须经仓储抽象。

### 5.2 Syncthing

- **假设**：办公室与家庭无缝衔接，**同一时刻仅一处写入**该数据目录。
- **实践**：换机前尽量正常退出应用；偶发冲突按 Syncthing 冲突副本规则处理。

### 5.3 标识与顺序

- **运行时主键**（会话、角色、用户、`turnId` / `receive.id`、API 预设与 Key 等）：**8 位 hex 短 id**（`server/src/short-id.ts`）；详见 `DOC/03-实现细节.md` **§6.7**。
- **`turnId`**：全局唯一、稳定引用；与 **`turnOrdinal`** 分离。
- **`turnOrdinal`**：当前选定路径上的「第 N 轮」；分支场景下「第 N 轮」仅对选定路径有意义。

## 6. 关键运行时链路（摘要）

1. 用户登录，颁发 JWT。
2. 打开会话：读取对应 `index.json` 与 chunk 链，按当前分支渲染。
3. 发送消息：解析 **chat** 绑定 → OpenAI 兼容请求 → **SSE** 推送到前端。
4. 组装提示词（**匹配与注入分离**，见 `DOC/03` **§14.9**）：**①** `userText` → memory（向量单次）+ history（近 N 轮）；**②** `userText + memory + history` → 资料库关键字匹配（递归 ≤2）；**③** 注入顺序：world → memory system → history system → userInput。
5. 若启用对话记忆：解析 **embedding** 绑定 → LanceDB 检索 `turnId` → 从 chunk 读完整 turn → 写入 **memory** system；**chunk 为权威**，Lance 可重建；**memory 检索不依赖 lore 结果**。
6. 若启用静态知识库 RAG（与资料库向量检索扩展）：解析 **embedding / rag / rerank** → 检索与重排（待与 §14 分表实现）。
7. 可观测性：请求日志、错误、耗时、token、memory 命中 `turnId`、RAG 命中明细（与 `DOC/02-需求说明.md` 一致）。

> **会话摘要**：由 **`plot-summary`** 插件经 **`completeWithContext`** 写入资料库，**不**再设全局 `summary` 功能绑定；见 `DOC/12`。

## 7. 插件系统

> **实现**：2026-05-26 起已落地最小宿主（分用户 registry、`settingsSchema` 设置页、Web/Server 动态加载）。详见 **`DOC/09-plugin-system-and-guidance-generate.md`**、`plugins/README.md`。

- 插件包全局安装在 **`data/plugins/<pluginId>/`**；启用与排序在 **`data/{userId}/plugin-registry.json`**；用户参数在 **`data/plugins/<pluginId>/{userId}/settings.json`**。
- 批量读写对话 turn 字段（非直接读 chunk）的规划见 **`DOC/10-plugin-conversation-host.md`**。
- 用户侧 preset 与 **包内 API 设定（可含密钥）** 并存：包内设定**仅作用于该插件**，见 `DOC/03-实现细节.md` §1.3（**插件 API 绑定与审计仍为 P2**）。
- 插件**不得**读写其它 API 配置；出站调用**一律经后端转发**。

## 8. 安全与多用户（内网）

- **认证**：JWT；API Key **不落前端明文**。
- **隔离**：数据路径按 **`userId`**（或等价租户键）分区，避免跨用户读取。
- **权限**：首版不做复杂 RBAC；依赖内网边界与应用内用户隔离；后续可按租户需求扩展。
- **插件**：包内密钥仅服务本插件；**禁止**插件访问他人 API 配置；**禁止**前端直连插件目标 URL（见 `DOC/03-实现细节.md` §1.3、§4）。

## 9. 运行时与部署

> **决策记录**：2026-05-29（本机单端口、不采用双端口生产）。**实现**：2026-05 起已落地（`scripts/run-prod.mjs`、`server/src/static-web.ts` 等）；用户说明见根目录 **`README.md`**。

### 9.1 决策摘要（仍有效）

- **主场景**：本机 / 内网单机；`!_start.bat` / `!_start.sh` → **`npm start`**，浏览器只打开 **`http://localhost:{serverPort}/`**（`config.yaml` 的 **`serverPort`**）。
- **生产形态**：`npm run build` 产出 `web/dist` + `server/dist`；Fastify 在 **`NODE_ENV=production`** 且 **`SERVE_STATIC=1`** 时托管静态资源、`/api/*` 与 **SPA fallback**（`server/src/static-web.ts`）。
- **开发形态**：**`npm run dev`**（`scripts/run-dev.mjs`：后端 `tsx watch` + 前端 Vite）；**`webPort` 仅开发**，Vite 代理 `/api`。
- **暂不采用**：对用户暴露双端口生产；前后端 **不同 origin 分离部署**（CDN + 独立 API 域）需 `API_BASE`、CORS 白名单、`exposedHeaders`、媒体 URL 绝对化等，**当前不排期**。

### 9.2 实现要点（与仓库一致）

| 项 | 说明 |
|----|------|
| 用户启动 | `!_start.bat` / `!_start.sh` → `npm start` → `scripts/run-prod.mjs` |
| 依赖 | `scripts/ensure-deps.mjs`：对 `package-lock.json` 与根/`server`/`web` 的 `package.json` 算 SHA-256 指纹，与 `node_modules/.deps-stamp.json` 比对；缺失或不一致则 **`npm install`**（英文日志）；`npm run dev` 同样调用 |
| 构建 | 根目录 `npm run build`（web + server）；缺 `dist`、git 版本与 `build-meta` 不一致，或倒计时内按 **B** 时自动 build |
| 静态与 API | `registerStaticWeb`（`@fastify/static` + `setNotFoundHandler` → `index.html`） |
| JWT | `server/src/auth-secret.ts`：`JWT_SECRET` 环境变量，或 `config.yaml` 的 `jwtSecret`，或 `data/.jwt-secret`（生产勿用 dev 默认值） |

### 9.3 仍兼容的运维方式（不替代默认路径）

- **反代单入口**：Nginx/Caddy 将 `/` → `web/dist`、`/api` → Fastify；对用户仍是一个域名/端口，与 §9.1 一致。
- **数据**：无论何种入口，`dataDir` 与 Lance 索引仍在 **API 进程所在机器**；Syncthing 同步的是数据目录，不是静态站。
