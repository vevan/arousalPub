# 实现细节

## 1. 建议数据模型

### 1.1 api_configs

- `_id`
- `userId`
- `name`（用户内唯一）
- `providerType`（MVP: openai-compatible）
- `baseUrl`
- `apiKeyEncrypted`
- `defaultModel`
- `capabilities`（如 chat/rerank/embedding）
- `isEnabled`
- `createdAt` / `updatedAt`

### 1.2 全局 API 默认（连接设置）

用户级出站 API 默认在 **`api-settings.json`** 维护，由设置页 **连接 / API 预设** 管理（`presets[]` + `activePresetId`）。

- **主对话 chat**：全局默认取 `activePresetId`；对话级可在 `index.json` → `apiPreset.chat` 覆盖。
- **RAG / Rerank 等**：在各自功能设置页绑定 `apiConfigId`（或对话 `apiPreset.rag` / `rerank`）；**不设**独立的全局「功能绑定」文档层。
- **插件 API**：在各插件 `settings.json`（`settingsSchema` 中的 `apiConfigId`）与可选对话 `apiPreset.plugins[pluginId]` 决定；见 `DOC/18` §3.8。

### 1.2.1 对话 API 设定（`index.json`）

会话根 **`apiPreset`** 含两类字段：

1. **功能键**（与 §1.2 对齐）：`chat`、`rag`、`rerank` 等，值为 `{ apiConfigId, modelOverride? }`。（**会话摘要**由 **`plot-summary`** 插件承担，不设 `summary` 键。）
2. **插件**：
   - **`plugins`**：对象，键为 **`pluginId`**，值为 `{ apiConfigId, modelOverride? }`；某插件可选用**与其它插件不同**的 API 预设。
   - **`plugin`**（可选）：本对话内**未在 `plugins` 中单独列出**的插件的**统一默认**（仍低于 `plugins[pluginId]` 优先级）。

**非插件键覆盖**：`resolved(key) = 对话.apiPreset[key] ?? 全局(featureType=key, "global")`。

**插件键解析（已知 `pluginId`）**：

`resolvedPlugin(pluginId) = 对话.apiPreset.plugins[pluginId] ?? 对话.apiPreset.plugin ?? 插件.settings.apiConfigId`

（`apiConfigId` 均指向已登记的 `api_configs`。）

### 1.2.2 对话级 API 覆盖（定案 · 已实现 2026-05-26）

> **2026-05 产品澄清**：对话可以覆盖「用哪条预设 + 采样参数」，**不能**在对话里自定义独立连接（`baseUrl` / `apiKey` / `apiKeyId`）。

#### 主 API（`chat`）

| 允许 | 禁止 |
|------|------|
| 从 **`api-settings.json` 已有 preset** 中选一条（`apiConfigId` = preset `id`） | 对话内填写 `baseUrl`、`apiKey`、`apiKeyId` |
| 覆盖 **非连接类参数**：`model`、`temperature`、`maxTokens`、`topP`、`topK`、DRY、`frequencyPenalty` 等 | 在对话里新建 preset 或导入连接 |
| `null` / 省略 → **继承全局 `activePresetId`** | 前端 body 传密钥 |

**解析**（服务端，`conversationId` 已知时）：

```text
preset = api-settings.presets[apiConfigId]   // 必须存在
params = merge(preset 默认字段, apiPreset.chat 覆盖字段)   // 仅白名单采样键
credentials = preset.baseUrl + preset.apiKeyId/keychain   // 始终来自 preset，对话不可改
```

**落盘**（`chats/{id}/index.json`）：

```json
{
  "apiPreset": {
    "chat": {
      "apiConfigId": "abc12345",
      "model": "gpt-4o-mini",
      "temperature": 0.9
    }
  }
}
```

仅写与全局/所选 preset **不同的参数字段**；清除覆盖：`PATCH { "apiPreset": null }` 或 `{ "apiPreset": { "chat": null } }`（实现时二选一并写死）。

#### Embedding（对话级）

与 memory / history 等设置同模式：**连接走全局** `user-preferences.embeddingApi`（`baseUrl` + Keychain），对话只稀疏覆盖 **模型参数**：

| 允许 | 禁止 |
|------|------|
| `embeddingApiSettings.embeddingModel` | `baseUrl`、`apiKey`、`apiKeyId` |
| `embeddingApiSettings.embeddingDimensions` | 对话内独立 embedding 网关 |
| `null` → 继承全局 | |

**落盘**：

```json
{
  "embeddingApiSettings": {
    "embeddingModel": "text-embedding-3-small",
    "embeddingDimensions": 1024
  }
}
```

已有 **`memoryEmbeddingModel` / `memoryEmbeddingDimensions`** 记录索引快照；改对话 embedding 覆盖且与快照不一致时，沿用现有 **记忆向量重建** 弹窗逻辑。

#### UI（已实现 · 2026-06-17 保存策略）

- **`ConversationContextSettings`** → **「API」** Tab：
  - **继承开关**：关 = 仅进入本地编辑（预填 global effective），**不 PATCH**；开 = 立刻 `PATCH` 清除覆盖
  - **离散字段**（preset、model、temperature、开关等）：短防抖（约 150ms）后 PATCH
  - **长文本**（自定义 JSON、DRY breakers、Embedding model 等）：**失焦**后 PATCH；与上次成功写入快照相同则跳过
  - **draft 保护**：编辑中 `syncFromProps` 不覆盖开关与字段；关 dialog 丢弃未保存 draft
- **会话插件设置**（`ConversationPluginSettingsPanel`）：schema 中 `string` / `text` / `textarea` 在 **`defer-text-commit`** 下失焦提交；离散字段短防抖 PATCH
- **全局 Embedding**（设置页）：`baseUrl` / `apiKey` / `embeddingModel` **失焦** `PATCH`；`apiKeyId` / `dimensions` 等仍短防抖
- **对话顶栏**（`ChatConversationView`）：effective preset/模型 pill、绑定提示词与世界书快捷入口、会话设置；详见 §11.2 `chat-header`。

### 1.3 插件：包内 API 设定、隔离与后端转发（V2）

**包内设定（可含密钥、URL、模型名、参数等）**

- 插件包可携带**仅作用于本插件**的连接信息（如 manifest / 插件私有配置区中的 `baseUrl`、`model`、参数、经宿主保管的 **apiKey**）。
- 此类设定**只能用于该 `pluginId` 执行过程中的出站调用**，**不得**视为对全局/对话/其他插件 API 预设的修改。
- **不得读写其它 API 配置**：插件运行时**禁止**调用宿主接口以枚举、读取或写入 `api_configs`、其它功能的连接设定、`index.json` 中的 `apiPreset`（除宿主明确提供的**只读自身配置**或**仅写本插件沙箱**的 API）。其它功能的 preset 对插件代码**不可见**（由宿主注入「已解析的连接」或仅允许无密钥的元数据）。

**与用户 preset 的关系**

- 用户通过 §1.2.1 **`resolvedPlugin(pluginId)`** 指向的 `api_config`，与包内设定是**两条来源**；具体「用户 preset 优先」还是「包内优先」可在插件 manifest 声明 **policy**（如 `connection: userPreset | embedded | userFirst`）。**缺省建议**：**用户已配置 `apiPreset.plugins[pluginId]` 时走用户 `api_config`**；未配置时允许使用包内设定（仍仅影响本插件）。无论哪种，**密钥不落前端**。

**出站请求**

- 插件对外部模型/工具 HTTP 的调用**必须经 Node 后端转发**（宿主 Runtime 使用服务端 `fetch` 等代发）；**禁止**浏览器端插件脚本直连第三方 URL 或持有明文密钥。

**前端 → 后端载荷（澄清）**

1. **调用系统已登记的 API**（使用 `api_configs` 中的条目）：  
   - 插件/前端**只提交 `apiConfigId`**（及与密钥无关的业务字段，如 `modelOverride`、消息体等业务参数）。  
   - **后端**根据 `apiConfigId` 查库、拼接 URL、鉴权头与完整请求体，再向模型商发起请求。

2. **调用自定义 API**（未使用或无法使用共享 `api_config` 条目的场景）：  
   - 插件/前端将**本插件作用域内的连接设定**提交后端（如 `baseUrl`、`model`、参数、路由模板等）；**密钥**以 **服务端已为该插件保存的秘密** 为准（通过 `secretRef` / 首次在服务端写入的加密字段等方式引用），**不在浏览器持久化明文**。  
   - **后端**校验请求归属 `pluginId`、在服务端合并密钥与头部，**拼接并提交**出站请求。

### 1.4 宿主与插件 `settings` 文件布局（建议）

**问题**：插件各自维护配置（含自定义 API 相关字段），是用**单一总 `settings.json`**，还是**主配置 + 每插件独立文件**？

| 方案 | 优点 | 缺点 |
|------|------|------|
| **单一总文件** | 备份/扫一眼只开一个文件 | 多插件共写易冲突；卸载/禁用插件时难清理；单文件膨胀 |
| **主 `settings.json` + `plugins/<pluginId>/settings.json`** | 与「每插件自治」一致；**Syncthing 冲突面小**；卸载可整目录删；便于版本与调试 | 主文件要维护**插件注册表**（启用、版本等）；每插件目录路径**固定约定**为 `plugins/<pluginId>/` |

**推荐（与当前「插件私有设定」一致）**：

- **宿主主文件**（例：`settings.json` 或 `config/settings.json`，相对应用/工作区根由实现选定）：**仅**放宿主级选项 + **插件清单**（`pluginId`、`enabled`、`semver` 等），**不**把各插件大段配置堆进主文件。插件侧用户配置路径**约定为** `plugins/<pluginId>/settings.json`（与现有 `plugins/<pluginId>/` 目录一致），一般**无需**在清单里再写 per-plugin 路径。
- **每插件**：`plugins/<pluginId>/settings.json` — 该插件的**全部**用户可改项，含自定义 API 的**非密钥**字段及对服务端密钥槽的 **引用**；与插件包内默认合并规则由宿主定义。

若坚持单文件，可退化为「主文件内嵌 `plugins: { id: { ... } }`」的 JSON，但在插件多时**不利于**增量同步与冲突隔离，**不推荐**作为默认形态。

### 1.5 绑定粒度：全局、对话、插件 id，而非「轮次」

- **配置分层**：**全局**（§1.2）→ **对话**（§1.2.1 功能键与 `plugin`）→ **`apiPreset.plugins[pluginId]`**（最细）；**不**在 `turns[]` 中存权威 preset。
- **轮次内可存快照**：`runtime` / `receive` 中的 `resolvedApiConfigId` 等仅作**审计与复现**；改全局或对话 preset 时**不要求**改写历史轮次。
- **备份设定**：**N**、**K** 绑定在**对话**元数据（与 `apiPreset` 并列）；见 **§8**。

- `POST /api-configs`

- `DELETE /api-configs/:id`
- `POST /api-configs/:id/test`（连通性测试）

**当前仓库（文件型 `api-settings.json` + `api-keys.json`，2026-06）**：

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` / `PUT` | `/api/settings` | 读写 API 预设列表与 `activePresetId` |
| `POST` | `/api/settings/presets/:id/test` | 两阶段连通性：`GET /models` + 对指定/当前 `model` 发最小 chat；Body 可选 `{ baseUrl?, model? }` |

对话级 API 覆盖写会话 `index.json` 的 `apiPreset`（含 `plugins`，见 §1.2.1），不经独立「功能绑定」REST。

## 3. 调用链路建议

1. 业务模块提交调用请求（含 `featureType`、当前 `conversationId`；若为插件则含 **`pluginId`**）。
2. **解析有效 preset**：
   - 若 `featureType !== plugin`：`apiConfigId` = §1.2.1 **非插件键覆盖**结果。
   - 若 **插件**：先 §1.2.1 **`resolvedPlugin(pluginId)`**；再按插件 manifest **policy** 与 §1.3 **包内设定**合并（仅影响本插件出站参数）；**实际 HTTP 仅由服务端发出**。
3. Config Service 根据最终解析结果取 `baseUrl`、密钥、默认模型（密钥永不下发浏览器）。
4. Runtime Service 发起 OpenAI 兼容请求（SSE 或非 SSE）或**代插件转发**至目标 URL。
5. 记录调用日志、耗时、token、RAG 命中明细；日志可记 `apiConfigId`、`pluginId`、是否使用包内连接。

## 4. 安全策略

- **对话 API Key / Keychain（2026-06 已实现，见 `DOC/25` §15）**：
  - `GET /api/api-keys`、`GET /api/settings`、`GET /api/user-preferences`（embedding）**不下发明文**；返回 `keyConfigured` 等元数据。
  - `PUT` / `PATCH` 支持省略 `key` / `apiKey` 时 **merge 保留磁盘**；`""` 表示清空。
  - `POST /api/chat`、`POST /api/models` 由 **`api-credential-resolve`** 按 `apiPresetId` / `apiKeyId` 读盘拼 `Authorization`；浏览器请求体**不带** `apiKey`。
  - `POST /api/api-keys/:id/reveal`：校验当前用户登录密码后一次性返回明文（仅 UI 临时展示，不写入 store）。
- 磁盘侧 `api-keys.json` / `api-settings.json` 仍为**服务端本地 JSON 明文**（未做 `apiKeyEncrypted`）；安全边界为**密钥不出浏览器**。
- 所有配置与绑定按 `userId` 隔离
- 删除 API 配置前检查是否存在 binding 引用（**已实现**：`GET/DELETE /api/settings/presets/:id`、`GET/DELETE /api/api-keys/:id`，见 §2.1）
- **插件**：包内密钥仅存入**插件作用域**；插件**不得**读写其它 API 设定；插件对外请求**仅经后端转发**（见 §1.3）

## 6. 对话文件存储（Chunk + Syncthing）

设计目标：单写者、办公室/家庭通过 Syncthing 同步数据目录；对话按固定轮数切成多个 chunk 文件；**导航与分叉信息可放在 chunk 的 `meta` 中**；会话与分支元数据由各级 `index.json` 承担（结构见 **§7**）。

> **实现状态（2026-06）**：**主路径已实现**——`chunk-chain.ts` 提供滚动 tail、`readAllTurns`、跨块 PATCH、`turnsPerFile` 可配置、超大单块迁移、`rebuildHeadTailFromLinks` / `syncChunkIndexIfDrifted`、`POST .../repair-chunk-index`、删空 tail 链式回退；详见 **`DOC/08-chunk-chain-implementation.md`**。  
> **分支**：字段与目录布局见本节 §6.1–§6.3；**memory 枚举 / 召回等服务端原语已落地，UI 与写入待做** — 实现对照 **`DOC/23-conversation-branches.md`**。

### 6.1 目录布局（示意）

```text
chats/
  chat.index.json
  chatA/
    index.json
    turn-000000-000099.json
    turn-000100-000199.json
    turn-000200-000299.json
    branch1/
      index.json
      turn-000100-000199.json
      turn-000200-000299.json
      branch1/
        index.json
        turn-000200-000299.json
    branch2/
      ...
  chatB/
    ...
```

说明：

- 文件名中的 `000000-000099` 表示**当前线性路径下的 turnOrdinal 闭区间**（仅辅助人类与排障；**权威顺序以 chunk 链与消息父子关系为准**）。主路径与分支子目录**共用同一命名公式**（`DOC/08` §2），**非**「分支第一个文件继承父目录序号」；详见 **`DOC/23` §1.5**。
- 分支子目录下可出现与父目录**同名**的 `turn-*.json`，内容属于该分支，不得跨目录混读。

### 6.2 Chunk 文件顶层结构

每个 chunk 文件建议固定为两层：**`meta`（导航与可选轻量元数据）** + **`turns`（本轮次数据）**。

```json
{
  "schemaVersion": 1,
  "meta": {
    "chunkId": "turn-000000-000099",
    "ordinalRange": { "start": 0, "end": 99 },
    "links": {
      "previous": "turn-000000-000099.json 或相对路径或 null",
      "next": "turn-000100-000199.json 或 null",
      "branches": []
    }
  },
  "turns": []
}
```

- `schemaVersion`：整文件格式版本，便于迁移。
- `meta.links`：**previous / next / branches 统一放在 `meta` 下**（与当前构思一致）；`branches` 见 6.3。
- `meta` 中可继续扩展：写入时间、最后修改客户端、校验和等（按需）。

### 6.3 `meta.links.branches`（分叉出口）

`branches` 为数组，每一项描述**从本 chunk 内某一 turn 分叉出去**的子树入口，建议最小字段：

- `forkTurnId`：分叉所在的 turn（权威引用，见 6.4）。
- `forkMessageId`（可选）：若一轮内有多条 message，精确到从哪一条消息分叉。
- `path`：子目录或子树根 `index.json` 的相对路径（相对当前 chunk 所在目录）。
- `label`（可选）：UI 展示用。

同一 chunk 可有多个分叉；子树目录内再维护自己的 chunk 链与 `index.json`。

**创建语义（定案 · `DOC/23` §1.4）**：**空分支 + 从下一轮继续** — 创建时仅注册 `branchN/` 与空 `index.json`，**不**复制 fork 点之后 turn、**不**在 fork 轮重写内容；分支首条新 turn 的 `turnOrdinal = forkTurnOrdinal + 1`；共享前缀由读路径合并，不写入分支磁盘。

### 6.4 `turnId` 与 `turnOrdinal`（必须区分）

| 字段 | 含义 | 说明 |
|------|------|------|
| `turnId` | 全局唯一、稳定标识 | 用于分叉引用、跨 chunk 指向、调试；**可与轮次序号解耦** |
| `turnOrdinal` | 当前选定路径上的第几轮 | 仅在同一条从根到叶的**线性路径**上有唯一「第 N 轮」含义；用于展示与 chunk 命名辅助 |

约定：

- UI 展示「第 N 轮」→ 使用 `turnOrdinal`（在已选路径上计算或缓存）。
- 存储分叉点、合并、引用 → 使用 `turnId`。
- `turns` 数组内顺序表示**本文件内时间顺序**；跨文件顺序由 `meta.links.previous/next` 串联。

### 6.5 一轮的语义：发送与接收

- **一轮**宜按 **「发送 → 模型 → 接收」** 界定，而不是简单按 OpenAI 消息里的 `role: user` / `role: assistant` 各算半轮。实际请求里「发给模型的一包内容」可能混合多种来源，映射为不同 `role`，但**产品语义上同属一次「发送」**。
- **发送侧**可包含：用户输入；**assistant 输入**（上一轮 API 返回且可编辑、或用户冒充助手写入，通常映射 `role: assistant`）；lorebook；history；用户的额外提示词（详见 §6.6）。
- **接收侧**：**当前轮次**模型**单次**返回的 completion（流式仍计一条接收）。上一轮已落盘的 assistant 正文若编辑后再参与请求，在**下一轮**归入发送侧的 assistant 输入，**不**重复记为新的「接收」。

### 6.6 提示词块组装（与 SillyTavern 管理器字段对齐）

对 lorebook、history、额外提示词等**可拆分块**，每块可配置：

| 字段 | 说明 |
|------|------|
| 姓名 | 显示名；逻辑 id 可另设。 |
| 身份 | 系统 / 用户 / 助手 → `system` / `user` / `assistant`。 |
| 触发器 | 在哪些生成类型下注入（正常、续写、AI 帮答、备选、重新生成、静默等）；未命中则不参与本轮。 |
| 位置 | **聊天中**：插在消息序列缝隙；**相对**：相对其它提示词或大模板锚点（设定前/后、历史前/后、用户输入前/后等）。 |
| 深度 | 含义依赖「位置」，见下。 |
| 排序 | 次要键：数值从低到高；仍并列时按 **助手 → 用户 → 系统**。 |

**位置 = 聊天中**（ST `injection_position: 1` / 本应用 `injectionPosition: 'chat'`）：以**最后一条 user 消息**为锚（含本轮 `userInput`；通常为栈底 user）。深度 **0** = 紧接该 user **之后**（AI 即将回复之前）；**1** = 其前一条消息；数值增大则继续向**更早聊天历史**回退。**不计入**其前的 system / 角色 / 世界观等固定块（实现：`resolveChatDepthInsertIndex`，`server/src/assemble-prompts.ts`；插入下界为 `historyStart`，避免误入顶部 system 区）。

**位置 = 相对**：先按锚点归区，再在区内用深度 + 排序；与「聊天中」的深度语义不同，实现须用**位置**字段区分。

**拼装**：触发器筛选 → 聊天中块与相对块分别按规则插入 → 同缝冲突先比深度、再比排序、再比身份次序 → 生成最终 `messages[]`。存储可保留逻辑块列表；**发往 API 的数组由规则生成**。preset chat-depth 条目与 **Author's Note** 共用上述 chat 深度锚点。

### 6.7 标识生成规则（性能优先）

- **8 位十六进制短 id**（`server/src/short-id.ts` / `web/src/utils/short-id.ts`：`generateShortId()`、`allocateShortId()`）：**会话目录名**、**角色卡 id / 文件名**、**用户 id**、**API 预设 id**、**API Key 别名 id**、**turnId / receive.id**、资料库/提示词内**用户新建**的分组与条目 id 等运行时主键。
- **语义化稳定 id**（字母数字与 `_`/`-`，见 `PROMPT_PRESET_ID_RE` / `LOREBOOK_ID_RE`）：仅用于**服务端种子**或产品约定常量，例如 `preset-default`、`lore-default`、`group-pre`；**禁止**前端在加载失败或空库时自动写入。
- **不推荐**作为默认路径：UUID v4、时间戳 slug、对每轮做 **SHA-256 等内容哈希**（分配与校验开销更大，且与现有路径/校验不一致）。
- **遗留数据**：磁盘上若仍存在旧版 UUID 或旧版 `key-…` 长 id 格式，读盘与引用解析仍兼容；**新创建**条目一律走短 id。
- **可选扩展**：若将来需要确定性幂等（同输入同 id），可另设辅助字段（如 `contentFingerprint`），**不**替代主键。

### 6.8 `turns[]` 中单轮结构、 `receives[]` 与 swipe（重新生成）

产品语义上，一轮 = 一次发送上下文 + 至少一条模型接收。**同一次发送**下多次「重新生成」产生多条候选 assistant 输出（swipe），**不**新开分支目录。

```json
{
  "turnId": "a1b2c3d4",
  "turnOrdinal": 12,
  "send": {},
  "receives": [
    {
      "id": "e5f6a7b8",
      "content": "…",
      "runtime": {
        "model": "…",
        "durationMs": 1200,
        "estimatedTokens": 4096,
        "completionTokens": 80,
        "resolvedFeature": {
          "featureType": "chat",
          "apiConfigId": "…",
          "source": "global"
        }
      }
    }
  ],
  "activeReceiveIndex": 0,
  "plugins": [
    {
      "pluginId": "weather",
      "payload": { "temperature": 20 }
    }
  ]
}
```

- **`send`**（可选）：结构化描述本轮请求前的输入包（逻辑块、主用户文本等）；可与 §6.6 逻辑块列表对应。
- **`receives[]`**：每次生成完成追加一条；**swipe** 即多条 `receive`。`activeReceiveIndex` 表示当前展示的候选（默认最后一条）。
- **`runtime`**（可挂在每条 `receive` 上）：审计快照；**非**对话级 API preset（见 §1.5）。`/api/chat` 落盘时写入 **`resolvedFeature`**（`featureType` / `apiConfigId` / `source`，可选 `model`），记录出站 chat 解析链结果。

**会话 debug 审计文件**（定案 · **`DOC/24` §3**）：**不**存放在 chunk 的每轮对象内。目标文件为 **`chat-audit.json`**（schema **v3**；与 chunk 并列），`entries[]` 含 **`messages`**（outgoing 最终态，含 regex outgoing 之后）、**`assembly`**（memory/lore/history 命中与裁切）、**`calls`**（chat / embedding）、**`performance`**（组装/上游/落盘耗时与流式 TPS；`auditDebug` 关闭时不计算）。`plugin.complete` 为 schema 预留，见 **`DOC/24` §3.6**。仅当会话 **`auditDebug.enabled === true`** 且 **`maxStored ≥ 1`** 时在 **`/api/chat` 落盘成功**后写入；关闭时**零写入**。

**当前实现（2026-06-09 · P0 已验收）**：**`chat-audit.json`**（`schemaVersion: 2`），条目含 **`messages`** + **`assembly`** + **`calls`**（chat + embedding）；开关 **`auditDebug.enabled` + `maxStored`**；**`GET .../chat-audit`**；**`/api/chat` 落盘成功后服务端自写**，不再依赖 **`debugPrompt`**。读盘兼容旧 **`chat-prompt.json`**；**`GET .../chat-prompt`** 只读别名（仅 `messages`）。删除整轮时按 **`turnId`** 移除 audit 条目。插件异步出站（如 `plot-summary` 自动摘要）**不在**本轮 audit 范围。Web：**`ChatTurnPromptDialog`** 三 Tab（提示词 / 组装命中 / 出站调用）；助手行「审计」入口；`calls` 展示出站/接收 tokens（上游 `usage` 或组装估算 / tiktoken）。定案与验收清单见 **`DOC/24` §3**。

**`turn-block-head`**（`ChatTurnBlock`）：章回 divider 下插件 slot 行；**常驻** `min-height`；`plot-summary` ▷/◁ 见 **`DOC/12` §7.2**。

**Swipe 与分支（branch）**：swipe 在同一 turn 内追加 `receives[]`；从某条消息**另起对话线**使用 `meta.links.branches` + 子目录（**空分支**，首条新 turn 为 `forkOrdinal + 1`，见 **`DOC/23` §1.4**）。显式「从某 receive 之后分叉」时可选写 `forkMessageId`；**不得**在 fork 轮重写 receive（与 swipe 区分）。分支实现步骤与代码索引见 **`DOC/23-conversation-branches.md`**。

**与当前仓库实现的对照（用户侧 swipe）**：上文 JSON 示例中 **`send`** 可为结构化对象；若产品需要「同一轮多次编辑用户稿 / 多候选 user 文本」，历史上可建模为 `sends[]` + `activeUserSendIndex`。**当前 Web + Node 存储已收敛为单条用户正文**：新写入统一为 **`send: { userText: string }`**（见服务端 `chat-storage` 与 API DTO）；磁盘上仍可能存在旧格式 **`{ sends, activeSendIndex }`**，读盘时通过 **`getTurnUserText`** 等兼容。前端无用户 swipe UI，仅可编辑当前 `turn.user`。**助手侧**仍为 **`receives[]` + `activeReceiveIndex`**（滑动切换、再生追加），与上文一致。

**Web 聊天滚动（`useChatSession`）**：

- 列表容器：`.chat-body.chat-scroll`；`scrollChatToBottom` 使用 `scrollTop = scrollHeight - clientHeight`。
- **触发**：新消息 / 流式增量、`loadMessages` 后、助手 **swipe 切换变体**（仅改 `activeReceiveIndex` 时 `nextTick` 后滚底）、重生成结束。
- **重生成 / swipe 再生**：在 `loadMessages` 前调用 `endRegeneratingUi()`（清除 `regeneratingTurnOrdinal` 与流式占位），避免按骨架高度滚底、正文与 swipe 工具栏出现后留空；`regeneratingTurnOrdinal` 置 `null` 时 watch 再滚一次兜底。
- **落盘与插件**：流式响应末包可带 `data: {"arousal":{"persist":{...}}}`；`readSseStream` → `onPersist` → `emitAssistantReplyPersisted`（`web/src/utils/chat-api.ts`、`useChatSession`）。后台标签对 **fetch / 主线程 / 布局滚动** 可能节流，落盘事件通常仍早于 `loadMessages` 完成。

**插件出站与资料库（规划）**：摘要等业务在**插件内**拼 `messages`；宿主提供 **`POST /api/plugins/:pluginId/complete`**（按 `apiConfigId` 转发，不走路由组装）、Lorebook **条目级** insert/patch、**`apply-order`**（按插件 layout 写 `order`）、会话 **`index.json` → `pluginSettings`**。详见 **`DOC/11-plugin-host-completion-and-lorebook.md`**、Historian 插件 **`DOC/12-plugin-plot-summary.md`**（标题可选用 `title-{startTurnNum}-{endTurnNum}` 后缀格式）。

### 6.9 与 `index.json` 的职责边界

- **Chunk 文件**：turn 正文、chunk 级导航（`meta.links`）。
- **各级 `index.json`**：会话树、chunk 清单、**对话级 `apiPreset`、备份 `N`/`K`**、分支注册；字段见 **§7**。

## 7. `index.json` 结构约定

### 7.1 `chats/chat.index.json`（会话列表）

每条会话条目建议包含：`conversationId`、`title`、**`userCharacterId`**（可选，用户 persona 角色卡 id）、**`characterId`**（可选，兼容单卡）、**`characterIds`**（可选，多卡绑定顺序）、`updatedAt`、`activeBranchPath`（可选，指向当前选中分支）。**`conversationId`** 为 **8 位十六进制**（如 `a1b2c3d4`），与 `chats/{conversationId}/` 目录名一致；生成与校验见 `server/src/conversation-id.ts`。服务端读写与会话根 `index.json` 的 **`characterId` / `characterIds`** 同步规则见 `server/src/chat-storage.ts`（**`resolvedCharacterIds`** 在 `characterIds` 为空时回退 **`characterId`**、`normalizeCharacterBindingsForWrite`）。

**列表快查冗余**（2026-06，服务端维护，供 `GET /api/chat/index` 与前端快查）：

| 字段 | 来源 | 说明 |
|------|------|------|
| `userName?` | 会话根 `index.json` | 宏 `{{user}}` 快照；绑 user 卡时通常与卡名一致 |
| `characterNames?` | `characters/index.json` | 与 **`characterIds`** 同序；卡已删时占位 **`已删除`** |
| `searchTags?` | 各绑定卡 + `userCharacterId` 对应卡的 `tags` | 去重合并；子串匹配语义同 §12 **`search`** |

- **写入**：`chatListEntryFromIndex` 拷贝 id/title/`userName` 等 → **`enrichChatListEntry`**（`character-storage.ts`）查角色索引填名与标签 → **`upsertChatListEntry`**。
- **失效**：`updateCharacterDocument` / `deleteCharacterFile` 后 **`refreshChatListEntriesForCharacter(id)`**（`chat-storage.ts`）重 enrich 所有引用该 id 的条目。
- **迁移**：`readChatList` 若存在绑卡但缺 `characterNames`（或应有 tags 却缺 `searchTags`），批量 enrich 后写回同一文件。
- **索引与磁盘对齐**：`readChatList` 先执行 **`reconcileChatListWithDisk`**——扫描 `chats/{conversationId}/index.json`，将索引中缺失的会话补进 `chat.index.json`（目录仍在则 API 不会漏条；用于修复列表文件与 `chats/{id}/` 不一致）。

### 7.2 会话根 `chats/<id>/index.json`

建议包含：

- `schemaVersion`、`conversationId`
- `title`、`characterId`、时间戳
- **`apiPreset`**：含功能键（`chat`、`rag`、`rerank` 等）及 **`plugins`**（按 `pluginId` 覆盖）、可选键 **`plugin`**（对话内插件总默认）；详见 §1.2.1；**仅写与全局不同的项即可**
- **`backupSettings`**：`everyNRounds`（N）、`maxKeptBackups`（K）
- **`auditDebug`**（可选）：`{ enabled, maxStored }` 控制 **`chat-audit.json`**；遗留 **`promptDebug.maxStored`** 读盘时映射为 `enabled`（见 **`DOC/24`** §3）
- **chunk 链**：`headChunkFile` / `tailChunkFile` 为**加速索引**；**权威以各 chunk 文件内 `meta.links` 为准**（见 §7.4）。不一致时以 **`meta.links` 重建索引**。
- **`branches[]`**（可选）：子分支入口列表，含 `forkTurnId`、`path`、`label`
- **`authorsNote`**（可选）：会话级作者注 `{ enabled, content, injectionDepth, role }`；`enabled` 仅在有非空 `content` 时为 true；组装时按 depth 注入 messages 栈（见 §15.8）

### 7.3 分支子目录 `…/branch*/index.json`

与 7.2 同形时可省略重复字段；子级 **`apiPreset` / `backupSettings`** 可覆盖父级，未写则继承。

### 7.4 一致性

- 对话级配置**不**写入 chunk 内 `turns[]` 作为权威源。
- **chunk 链**：写入新 chunk 时更新 `meta.links`；`index.json` 中的 head/tail 与链**应同步**；若发现漂移，以磁盘上 chunk 的 `meta.links` 为准修复索引。
- 删除 `api_configs` 中某条前，须检查 **`activePresetId`**、**各功能页绑定**、**各对话 `apiPreset`（含 `plugins` 内条目）** 引用。

## 8. 备份与恢复

### 8.1 目标

- **防丢失**、**可恢复**；**Syncthing** 负责多机热同步；**产品内冷备**由 **§8.8 启动全量 zip** 承担（`config.yaml` 间隔天数 + 保留份数）。**§8.4 对话轮次增量**已**无限期延后**，不实现。

### 8.2 备份范围

以数据根目录（如 `data/`）为单元，按用户子目录 **`data/{userId}/`**（默认 `default-user`）存放：`chats/`、`prompts/`（`index.json` + 各 `preset-*.json`）、`characters/`、`api-settings.json`、`api-keys.json`、`lorebooks/` 等。排除 `node_modules`、构建产物、可重建缓存。备份介质须与生产数据同等控制权限；敏感项含 API Key、密码哈希。路径约定见 `data/README.md`。

### 8.3 一致性前提

单写者前提下，备份/恢复前仍建议**停应用或确保刷盘**，避免半写文件。

### 8.4 对话轮次增量备份【无限期延后 · 不实现】

> **状态**：架构曾预留 `backupSettings`（`everyNRounds` / `maxKeptBackups`）于对话 `index.json`（§7.2），默认 `0`（关闭）。**不与 §8.8 并行开发**；若未来需要「单对话时间点恢复」，优先考虑**手动导出对话**或对话检查点，而非每 N 轮自动 subtree 包。

- **Syncthing**：仍负责多机镜像；误删风险靠 **§8.8 冷备** + Syncthing **ignore `backups/`** + 恢复流程（§8.5）。
- ~~对话轮次触发增量~~：从路线图移除，**无限期延后**。

### 8.5 恢复流程

1. 停止应用。2. 当前数据根可改名为 `data.broken-<时间戳>`。3. 从选定备份还原。4. 启动验证。5. 使用 Syncthing 时，恢复宜暂停同步或单方权威副本后再同步。

### 8.6 安全（内网）

目录权限控制；离线拷贝可加密。多用户时可按 `userId` 子树备份。

### 8.7 实现衔接

- 固定 `DATA_ROOT`；可选 `backup.bat` / `backup.sh`；**主路径 §8.8 产品内冷备**。
- ~~对话轮次增量 subtree~~：已无限期延后（§8.4）。

### 8.8 全量冷备 zip（`data/backups`）【P0 · 产品内主备份】

> **状态**：**已实现**。多机同步仍靠 **Syncthing**；本节为 **本机时间点快照**，不提供 HTTP 下载。

#### 8.8.1 目标

- 在 **`{dataDir}/backups/`** 落盘全量 zip（含 `chats/`、`memory/` Lance、`lorebooks/`、`api-keys.json` 等；**排除** `backups/` 自身）。
- **Syncthing** 对共享的 `data` 文件夹 **Ignore `backups`**，避免大 zip 参与 PC ↔ NAS ↔ 树莓派 ↔ 笔记本同步。
- 与「聊完关服务、Syncthing 搬运生产数据」分工：冷备只留本机，不作多机镜像。

#### 8.8.2 触发策略（定案）

| 项 | 定案 |
|----|------|
| **何时检查** | **服务启动时**（`listen` 后异步），不依赖浏览器是否打开 |
| **条件** | 距上次**成功**备份超过 **N 天**（`config.yaml` 如 `backupIntervalDays`），或从未备份 |
| **UI** | 备份进行中 **冻结 Web**（全屏进度，类似 `authPhase === 'loading'` / 记忆重建 SSE）；**冻结写入**（变更 `data` 的 API 返回 `503 backup_in_progress`） |
| **不提供** | 浏览器下载 zip；备份文件仅存在于 `data/backups/` |

#### 8.8.3 配置字段（已实现）

| 字段 | 说明 |
|------|------|
| `backupEnabled` | 默认 `true`；`false` 则跳过启动时自动冷备 |
| `backupIntervalDays` | 超过该天数则在下次启动时触发（默认 7） |
| `backupMaxKept` | `backups/` 内最多保留 K 个 zip，超出删最旧 |
| `backupRetryHours` | 失败后暂缓重试（默认 24） |

记录上次成功时间：`data/backups/backup-manifest.json`（`lastSuccessAt`、`file`、`bytes`）；避免依赖半截 zip 的 mtime。运维与 Syncthing 边界见 **`data/README.md` §备份**。

#### 8.8.4 实现要点

- **服务端**：`server/src/data-backup.ts` — 遍历 `DATA_DIR` 流式 zip（**排除** `backups/`）→ `backups/backup-<时间>.zip`（先 `.tmp` 再 rename）→ `backup-manifest.json` → 滚动删超 K 份。
- **配置**：`config.yaml` — `backupEnabled`、`backupIntervalDays`（默认 7）、`backupMaxKept`（默认 5）、`backupRetryHours`（默认 24，失败后暂缓重试）。
- **API**：`GET /api/backup/status`（**免 JWT**）：`running`、`filesDone`、`filesTotal`、`lastSuccessAt`、`lastError`。
- **前端**：`initAuth` 前轮询 status；`running` 时全屏「备份中…」，完成后进入登录/应用。
- **写锁**：备份期间 `acquireMaintenanceLock('data_backup')`；变更 `data` 的写 API 返回 **503** `backup_in_progress`。
- **触发**：`app.listen` 后异步检查；距上次**成功**超过间隔天数或从未成功则执行。

#### 8.8.5 与 §8.4（已延后）的区别

| | §8.4 对话增量（延后） | §8.8 全量冷备 |
|--|----------------------|----------------|
| 状态 | **不实现** | **已实现** |
| 触发 | 每对话满 N 轮 | 启动且超过 N **天** |
| 范围 | 单对话 subtree | 整棵 `data` |
| 落盘 | — | `backups/*.zip` |

#### 8.8.6 多机边界（Syncthing）

- **同步**：`data/{userId}/` 生产数据、`.jwt-secret`、`.data-encryption-key`（各实例须**相同** DEK）。
- **忽略**：`backups/` 整个目录（`.stignore` 加一行 `backups`）；冷备 zip 仅本机时间点快照，不作多机热镜像。
- **恢复**：§8.5 + **`data/README.md` §恢复流程**；恢复时宜暂停 Syncthing 或指定权威副本。

## 9. 前端国际化（Web）

MVP 在 **Web 客户端**提供界面语言切换，与右侧抽屉中的 **API 连接**（密钥、模型等）分离：后者仍仅存在于「连接」面板。

### 9.1 依赖与文件

- **路由**：`vue-router`，设置页路径为 `/settings`（顶栏「设置」进入）。
- **文案**：`vue-i18n`（Composition API，`legacy: false`）。
- **翻译表**：`web/src/locales/en.json`、`web/src/locales/zh.json`，键按域分组，例如 `app`（壳与导航）、`chat`（对话页）、`conn`（连接侧栏）、`settings`（设置页）、**`characters`（角色库页）**、`prompts`（提示词库）。新增可见文案时**同步维护**两种语言，避免缺键回退不可控。

### 9.2 默认语言与回退

- 用户偏好保存在浏览器 **`localStorage`**，键名 **`arousal-locale-pref`**，取值 **`auto`** | **`en`** | **`zh`**。
- **`auto`（默认）**：根据 **`navigator.languages`**（若为空则用 **`navigator.language`**）逐项匹配：
  - 语言标签（大小写不敏感）以 **`zh`** 开头 → 使用 **`zh`**（简体界面）；
  - 以 **`en`** 开头 → 使用 **`en`**；
  - **其余语言或未取到可用值** → **回退 `en`**（与 `vue-i18n` 的 `fallbackLocale` 一致）。
- **`en` / `zh`**：强制对应语言，不随浏览器列表变化。

### 9.3 与 DOM、日期的衔接

- **`document.documentElement.lang`** 随当前界面语言更新为 **`zh-CN`** 或 **`en`**（辅助无障碍与字体/排版习惯）。
- 侧栏等处 **日期时间展示** 使用 **`Intl`**（如 `toLocaleString`），与当前界面语言一致（实现上按 `zh` / `en` 映射为 `zh-CN` / `en-US`）。

### 9.4 浏览器语言变更

- 当偏好为 **`auto`** 时，监听窗口 **`languagechange`**，重新解析浏览器语言并刷新 `vue-i18n` 与 `lang`，无需整页刷新。

### 9.5 文案中的字面量花括号（提示词宏名）

**vue-i18n v11** 默认用 **Message Format** 编译 `locales/*.json`：`{name}` 表示插值占位符（如 `chat.errors.requestFailedStatus` 里的 `{status}`）。

若文案需要**原样显示**服务端提示词宏（双花括号），例如 `{{user}}`、`{{char}}`，**不可直接写入 JSON**，否则会编译失败（如 `Not allowed nest placeholder`、`Unterminated closing brace`），并连带导致依赖 `$t` 的弹窗渲染异常。

**约定（v11.3+，本仓库 vue-i18n 11.4.x）**：在 JSON 中对每个 `{` / `}` 使用反斜杠转义（JSON 内写双反斜杠）：

| 界面显示 | `en.json` / `zh.json` 中写法 |
|----------|-------------------------------|
| `{{user}}` | `\\{\\{user\\}\\}` |
| `{{char}}` | `\\{\\{char\\}\\}` |
| `<memory>` 等 XML 标签名 | `{'<'}memory{'>'}`（字面量插值，见 [Literal interpolation](https://vue-i18n.intlify.dev/guide/essentials/syntax.html#literal-interpolation)） |

示例：

```json
"createDialogHint": "…资料库。\\{\\{user\\}\\} 会保存为用户卡名称快照；…"
```

**不要**使用旧式单引号拼接（如 `'{'{'user{'}'}'}`），容易多写未加引号的 `}` 而触发 `Unterminated closing brace`。

**维护**：新增/修改含宏名的 UI 文案时，`zh` / `en` 同步转义；改完后可在仓库根用 Node 快速校验编译（可选）：

```bash
node --input-type=module -e "import { createI18n } from 'vue-i18n'; import zh from './web/src/locales/zh.json' with { type: 'json' }; const i18n = createI18n({ legacy: false, locale: 'zh', messages: { zh } }); console.log(i18n.global.t('conversationList.createDialogHint'));"
```

参考：[Message Format Syntax — Escape Sequences](https://vue-i18n.intlify.dev/guide/essentials/syntax.html#escape-sequences)。vue-i18n 已升级 v11（2026-05-29）。

## 12. 角色库（当前仓库已实现）

### 12.1 存储路径与单文件结构

- **目录**：**`{DATA_DIR}/{userId}/characters/`**（`getCharactersDir()`；`userId` 由 `server/src/user-context.ts` 解析，默认 `default-user`；`DATA_DIR` 见 `config.yaml#dataDir` / 环境变量，默认仓库根下 `data/`）。
- **主存单卡文件**：**`{id}.png`**（`id` 为 8 位 hex，与会话 id 同规则）。PNG 内嵌 SillyTavern 约定 **`chara`** 元数据：`tEXt` 或 `zTXt` 关键字 **`chara`**，值为 **Base64(UTF-8 JSON)**，JSON 形态为 **`{ spec: 'chara_card_v2', spec_version: '2.0', data: { ... } }`**。
- **规范化**：`server/src/character-png.ts` 中 **`normalizeTavernCardV2Data`** 在读写 `chara` 时补全 TavernCardV2 **`data`** 约定字段（字符串域、`alternate_greetings[]`、`tags[]`、`creator`、`character_version`、`extensions`（默认 `{}`）、可选 **`character_book`**），并透传其它未知键（跳过 `spec` / `spec_version` 以免污染扁平 `card`）。**组装注入**见 **`DOC/27`**（P2，当前仅存储透传）。
- **默认立绘（非用户数据）**：**`server/assets/characters/default-avatar.png`**，由 **`resolveDefaultAvatarPath()`** 解析；新建无图时用于生成首张 PNG。
- **列表索引**：`characters/index.json`（`schemaVersion: 1`，`entries[]` 含列表摘要字段）；创建 / 导入 / **PATCH** / 删除后同步；缺失或与磁盘 `*.png` 数量不一致时从磁盘 **重建**。
- **API 文档顶层**（`CharacterStoredDocument`，JSON 响应）：`schemaVersion: 1`，`id`，`importedAt`，`updatedAt`，**`card`**（与上述 **`data`** 字段一致的扁平对象，供前端与 PATCH 使用）。

### 12.2 「已使用 / 未使用」筛选

- **已使用**：`usedInConversationCount > 0`，计数来自 **`readChatList`** 聚合：遍历当前用户 **`chats/chat.index.json`** 中每条会话的 **`characterId` / `characterIds`**（经 **`resolvedCharacterIds`** 解析，含 legacy 单卡），对每个角色文件 `id` 累加引用次数。
- **未使用**：计数为 0。删除会话后无需改角色文件；刷新列表即重算。
- **侧栏计数 `filterCounts`**：`GET /api/characters` 响应 `{ all, used, unused }`，在 **搜索过滤之后、used/unused 筛选之前** 统计；切换「全部 / 已使用 / 未使用」时 **「全部」数字不随当前筛选项变化**（始终反映当前搜索下的全集规模）。

### 12.3 HTTP API（Fastify）

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/characters` | Query：`offset`、`limit`（默认 24，最大 100）、`search`、`filter`（`all` \| `used` \| `unused`）、**`sort`**（`recentChat` \| `recentUpdate` \| **`name`** \| `usageCount`，默认 **`name`**）、**`order`**（`asc` \| `desc`，默认 **`asc`**）。响应：`items`、`total`、**`filterCounts`**、`offset`、`limit`、`hasMore`。名称排序见 **`server/src/character-name-sort.ts`**（拉丁开头 → 汉字拼音 → 其它）。 |
| `POST` | `/api/characters/import` | Body：整卡 JSON 或 `{ card }`（及 `normalizeImportCard` 支持的 `data` 包裹形态）。规范化后写入新 **`{id}.png`**。响应：`{ ok, id }`。 |
| `POST` | `/api/characters/import-png` | **multipart**，字段 **`file`**（PNG 字节）。解析 `chara` 后 **原样拷贝**为新 `{id}.png`（分配新 8 位 id）。响应：`{ ok, id }`。 |
| `POST` | `/api/characters` | **JSON**：`cardFromNewCharacterForm` 所需字段（含 `creator`、`alternate_greetings`、`extensions` 等可选）。**或 `multipart/form-data`**：字段 **`payload`**（JSON 字符串）+ 可选文件 **`portrait`**（PNG）。写入 **`{id}.png`**。响应：`{ ok, id }`。 |
| `GET` | `/api/i/:token` | **公开**立绘/头像 PNG。`:token` 为 **`shared/portrait-media-token.ts`** 编码的 base64url（9 字节：`userId(4B) + imgId(4B) + kind(1B)`；`kind=0` 角色立绘）。Query 可选 **`size`**（`xs`/`s`/`m`/`l`/`xl` → 最长边 64/128/256/512/1024 px，sharp 缩放仍输出 PNG）、**`v`**（cache bust，不参与解码）。无 `size` 返回原图。支持 **`ETag` / `304`**。 |
| `GET` | `/api/characters/:id/export-png` | 下载 **`{id}.png`** 字节；`Content-Disposition: attachment`，文件名由 **`buildCharacterExportFilename`**（`card.name` slug，冲突加时间戳）。 |
| `GET` | `/api/characters/:id/export-json` | 下载 ST v2 整包 JSON（**`buildStV2CharacterExport`**）；`Content-Disposition: attachment`。 |
| `POST` | `/api/characters/:id/portrait` | **multipart**，字段 **`portrait`**（PNG）。保留当前 `card`，替换图像并重写 `chara`。响应：完整文档 JSON。 |
| `PATCH` | `/api/characters/:id` | Body：`{ card: { ... } }`，**浅合并**后规范化并写回 **`{id}.png`**，更新 `updatedAt` 与 `index.json`。成功响应为完整文档 JSON。 |
| `GET` | `/api/characters/:id` | 返回完整存储文档（含 `schemaVersion` 与 `card`）。`:id` 须为 8 位 hex（`isValidShortId`）。 |
| `DELETE` | `/api/characters/:id` | 删除 **`{id}.png`** 并更新索引。 |

### 13.5 资料库匹配语料与递归级联（规划）

与 **§14.9** 组装前管线一致：**先**用 `userInput` 得到 memory / history，**再**用合并语料匹配资料库（非仅 `userText`）。

#### 13.5.1 扫描语料 `scanCorpus`

```text
scanCorpus = buildScanText(userText, memoryText, historyText)
```

| 组成部分 | 来源 | 说明 |
|----------|------|------|
| `userText` | 本轮输入框 | 必选 |
| `memoryText` | §14 memory 序列化正文（或仅其中的 user/assistant 纯文本） | 可选纳入 |
| `historyText` | §14 近 N 轮序列化正文 | 可选纳入 |

- **会话级 `loreScanScope`**（规划）：
  - **`userOnly`**：仅 `userText`（等同当前实现）。
  - **`userMemoryHistory`**（**推荐默认**）：三者合并后扫关键字；便于「用户未提、但近期对话或召回记忆里出现」的设定名触发。
- **裁剪**：`maxScanChars`（如 8k～16k）；`historyScanTurnCount`（≤ 注入用 N，如扫 key 只用最近 6～10 轮）以降低误触发与 CPU。
- **可选（规划）**：`loreScanUserLinesOnly` — memory 仍可从 `<memory>` XML 内只取 `<user>` 文本；近期 history 扫描语料为 **纯文本**（`turnsToHistoryScanPlainText`，与注入的 user/assistant 链同源）。

#### 13.5.2 递归级联（`maxDepth = 2`）

对齐 SillyTavern「recursive scanning」语义，**硬顶深度 2**（产品可调 0～3，默认 **2**）：

| 轮次 | 扫描文本 | 动作 |
|------|----------|------|
| 0 | `scanCorpus` | 关键字 / `constant` → 命中集 S₀ |
| 1 | `scanCorpus` + S₀ 各条 `content` 拼接 | 再匹配 → S₁ |
| 2 | 上轮文本 + S₁ 新增 `content` | 再匹配 → S₂（**最后一轮**） |

**每轮必须**：

- 已命中 **`entry.id` 去重**（防 A↔B 互指死循环）。
- **`loreMaxMatchedEntries`**（如 16）与 **`maxLoreTokens`**（tiktoken 截断整段 `<lores>`）。
- `constant` 条目仍按现规则参与（通常第 0 轮即纳入）。

**禁止**：

- 递归每一轮调用 embedding / Lance（资料库级联**仅规则**，不走向量多跳）。
- 用 lore 命中结果再触发第二轮 **memory** 检索（除非独立开关且全局仅多 **1** 次 embedding）。

#### 13.5.3 默认参数（规划）

| 字段 | 默认 | 说明 |
|------|------|------|
| `loreRecursiveEnabled` | `true` | 关则 `maxDepth=0` |
| `loreMaxRecursionDepth` | `2` | 硬顶 `3` |
| `loreScanScope` | `userMemoryHistory` | 见 §13.5.1 |
| `loreMaxMatchedEntries` | `16` | 单次组装最多条目 |
| `maxLoreTokens` | （实现时与 context 联动） | 整段 world 注入上限 |

## 14. 对话记忆（History / Memory + LanceDB）【设计已定 · 部分已实现】

### 14.1 背景与目标

- **问题**：`contextLength`（组装时 `maxTokens`）超限后，`assemble-prompts` 会从 **history 区间**按条删除最旧消息（`droppedHistoryCount`）。被删的是 **prompt 副本**；完整轮次仍在 **`chats/{conversationId}/` chunk** 中。
- **目标**：在上下文有限时，用 **hybrid 检索**（向量语义 + BM25 词法，RRF 融合）找回与当前用户输入相关的**较早轮次**，与 **近期 N 轮明文**分工注入，避免模型混淆「连续对话」与「回忆片段」。
- **原则**：
  - **chunk `turns[]` 为权威**；LanceDB 只存向量与 `turnId` 等指针，命中后按 `turnId` 从 chunk **读取完整 turn**。
  - **不把向量召回轮伪装成连续对话**：Memory 用 **一条 `system` + `<memory>`**；**近期 History** 用 **`user` / `assistant` 消息链**（见 §14.3），与本轮 `userInput` 的 role 语义一致。
  - **仅当前 active 分支**参与索引与检索（`turnId` 稳定、分支路径过滤）。

### 14.2 概念：History vs Memory

| 槽位 | 含义 | 数据来源 | 在 `messages[]` 中 |
|------|------|----------|-------------------|
| **History（近期历史）** | 当前轮之前的 **最近 N 轮**完整对话 | 沿当前分支 chunk 链取 turn，`turnOrdinal` 小于本轮 | **`history` 分组内多条** `user` / `assistant`（`turnsToHistoryMessages`，见 `turn-memory-xml.ts`） |
| **Memory（记忆召回）** | 与本轮 query **语义 + 词法相关**、且不在近期 N 轮内的旧轮 | Lance hybrid → `turnId[]` → chunk 解析完整 turn | **1 条 `role: system`**，正文为 `<memory>…</memory>` |
| **UserInput（本轮）** | 用户正在发送的输入 | 组装上下文 `userInput` | 现有 **`boundUserInput` / user 槽**（不变） |

**「完整 turn」**（与 `turnsToHistoryMessages` 一致）：

- `send` → `getTurnUserText(turn)` → **`role: user`**
- `receives[activeReceiveIndex].content`（trim 后非空）→ **`role: assistant`**
- **默认不包含**：未选中的 swipe 候选、`reasoning`（除非产品另开开关）

### 14.3 组装顺序与预设扩展

推荐在预设 `groups[].order` 中的相对顺序（在角色 / 资料库之后、用户输入之前）：

```text
… character / world(lore) …
→ memory（system，远期检索，<memory> XML）
→ history（近期 N 轮：user, assistant, user, assistant, …）
→ boundCharacterPostHistory（若有）
→ … 聊天中(injection) …
→ userInput（本轮 user）
```

**预设绑定槽**：

| 绑定槽 | 分组建议 | 注入 |
|--------|----------|------|
| `boundMemory` | `world` 后或独立 `memory` 分组 | 一条 system，内容为 `ctx.memoryText` |
| `boundRecentHistory` | `history` 分组 | 与默认相同：展开 `ctx.history` 为多条 `user` / `assistant`（`assemble-prompts` `injectRecentHistoryMessages`） |

`AssembleContext`（`assemble-prompts.ts`）：

- `history?: ChatMessage[]` — 近期 N 轮，由 `memory-pipeline` → `turnsToHistoryMessages` 填入
- `memoryText?: string` — 已格式化的 `<memory>…</memory>`
- `memoryTurnIds?: string[]` — 可观测 / 调试
- 会话级 **`historySettings`**（N）、**`memorySettings`**（TopK 等，存会话根 `index.json` 或全局 user-preferences）

**序列化**：

- **Memory（已实现）**：`<memory>` XML，与资料库 `<lores>` 风格一致（`formatMemoryXml`）。
- **History（已实现）**：**不再**向模型注入 `<history>` 块；`formatHistoryXml` 仍保留于 `turn-memory-xml.ts`（调试/兼容），组装主路径用 **API 式消息链**。
- **Lore 扫描**：近期对白用 `turnsToHistoryScanPlainText`（与 `history[]` 正文同源），与 memory XML 剥标签后的文本一并传入 `buildScanText`。

```xml
<memory>
  <turn ordinal="{turnOrdinal}" correlation="0.82">
    <user>…</user>
    <assistant>…</assistant>
  </turn>
</memory>
```

**去重**：若 `turnId` 已在近期 N 轮内，**不得**再出现在 memory 中。memory 检索可加 `turnOrdinal < min(近期窗口)` 过滤。

### 14.4 运行时链路（发送前 · 摘要）

完整分阶段说明见 **§14.9**。摘要：

1. 解析分支、本轮 `turnOrdinal`、`userText`。
2. **Memory**：以 `buildMemoryRecallQuery` 为 query 做 embedding + Lance **单次 hybrid** TopK（**不递归**）；FTS 与 embedding 共用同一 query 文本。
3. **History**：近 **N** 轮 turn → `recentHistoryMessages`（确定性，非向量召回）。
4. **Lore**：`scanCorpus = buildScanText(userText, memoryText, recentHistoryScanText)` → 关键字匹配 + **递归深度 ≤2**（§13.5）→ `ctx.world`。
5. **`assemblePrompts`**：注入顺序见 §14.3（world → memory → history 消息链 → userInput）。
6. **Token 裁切（§14.4.1）**：对完整 `messages[]` 估算 token；超出 `contextLength` 时进入统一裁切循环（见下节）。`assemblePrompts` 在会话组装路径设 `skipInternalBudgetTrim: true`，不再单独裁 history 区间。

#### 14.4.1 Token 预算裁切（定案）

**不可裁切**（始终保留）：

| 类别 | 说明 |
|------|------|
| 提示词预设 | normal / character / chat-depth 等相对与聊天深度条目 |
| 角色 / 用户 persona | bound character 卡体、system、post_history |
| 资料库 **恒定（constant）** | `triggerMode=constant` 条目，单独收集为 `constantLoreGroups` |

**可裁切**（逐条删除；**每删一条即重算 token**，直到 ≤ `contextLength` 或无可删项）：

| 顺序 | 槽位 | 删除策略 |
|------|------|----------|
| 1 | **匹配 lore** | keyword / vector 命中；**以相关性为主**：vector / hybrid **低 RRF 分**优先，keyword **低 priority** 优先（**不按**条目 `order` / 时间） |
| 2 | **memory** | hybrid 召回 turn；候选池内 **按 `turnOrdinal` 删最早**（**不按** RRF 分删；score 仅用于 TopK 召回与 audit） |
| 3 | **history** | 近期窗口 user/assistant 链；**最旧一条**优先 |

**下限**：匹配 lore、memory、history 各类 **`minRetain` 至少保留 N 条**（默认均为 1；0 表示允许全部丢弃）；某类已达下限时跳过该类，改按 `trimOrder` 尝试下一类。

**配置**：全局 `user-preferences.budgetTrim`；会话 `index.json` → `budgetTrimSettings` 稀疏覆盖（`null` 清除 = 继承全局）。字段：`trimOrder`（`lore` | `memory` | `history` 排列）、`minRetain.{lore,memory,history}`。

**实现**：`server/src/budget-trim-settings.ts` + `prompt-budget-trim.ts`；入口 `buildConversationOutboundMessages`。

**前端**：设置页 Tab「预算裁切」（`BudgetTrimSettingsPanel`）；会话 **`ConversationContextSettings`** 独立 Tab（可继承全局）；`assemble-messages` 预览对话框展示 dropped 计数。

**响应字段**：`assemble-messages` 返回 `droppedLoreCount`、`droppedMemoryCount`、`droppedHistoryCount`。

> 注：旧方案「memory 槽占 contextLength 18% 预裁」（`memory-token-trim`）已废弃，由上述统一循环替代。

**索引写入时机**（增量，已实现）：

| 事件 | 动作 |
|------|------|
| 新轮落盘 | insert / upsert by `turnId` |
| PATCH 改稿、再生、swipe 切换 active receive | update（同 `turnId` 覆盖） |
| 删轮 / 删会话 / 弃用分支子树 | delete by `turnId` 或会话+分支 |

**索引文本**：默认 `userText + "\n\n" + assistantText`。

**Lance 行字段（v2）**：`turnId`（PK）、`turnOrdinal`、`branchPath`（主路径 `""`）、`chunkFileName`（basename）、`corpus`（与 `turnEmbeddingCorpus` 相同，供 BM25 FTS）、`vector`。`memoryEmbeddingModel` / `memoryEmbeddingDimensions` 在会话 `index.json`。旧 `mem_*` 多表不兼容；**无 `corpus` 列的旧 `turn_memory` 须重建索引**。

#### 14.4.2 Hybrid 检索（Memory + 资料库 vector）【定案 · 2026-06-12】

**原则**：

| 项 | 定案 |
|----|------|
| **keyword 资料库** | **不变**：`keys[]` 对 `scanCorpus` 子串匹配 + 递归；**不**走 BM25 |
| **Memory 召回** | Lance **hybrid**：`vectorSearch` + FTS(`corpus`) + **`RRFReranker`**；query 文本 = `buildMemoryRecallQuery` |
| **资料库 vector 召回** | 同上；FTS 列 `text` = `lorebookEntryEmbeddingCorpus`；query 文本 = `scanCorpus`；仅 `triggerMode=vector` 条目 |
| **FTS 分词（用户级）** | `user-preferences.json` → **`hybridFts`**：`zh-ngram`（默认）、`en`、`zh-jieba`；`zh-jieba` 另选词典规格 `small` / `default` / `big`。详见 **§14.4.3** |
| **召回 vs 裁切** | **召回**按 RRF `_relevance_score` 取 TopK；**memory 预算裁切**在候选池内删 **最早 `turnOrdinal`**；**lore 预算裁切**仍按 **相关性**（vector 低分 → keyword 低 priority） |
| **降级** | FTS 索引缺失或 hybrid 失败时，**回退**纯 `vectorSearch`（`1/(1+_distance)` 作 score） |

**资料库 Lance 表 `lore_entries`**：在既有 `entryId` / `lorebookId` / `vector` 上增加 **`text`** 列 + FTS 索引；重建向量索引时一并写入。

**实现**：`server/src/lance-hybrid-search.ts`（FTS 配置与 hybrid query）；`hybrid-fts-dict.ts` / `lance-language-model-context.ts`；`memory-store.ts` / `lorebook-vector-store.ts`；`prompt-budget-trim.ts`（memory 按 ordinal 删）。

#### 14.4.3 Hybrid FTS 分词与词典目录【定案 · 2026-06-14】

Memory 与资料库 vector 的 BM25 共用同一套**用户级**分词配置；切换 profile 或 `zh-jieba` 规格后须**重建**相关 Lance FTS 索引（设置页确认弹窗会提示）。

**配置落盘**（`data/{userId}/user-preferences.json`）：

```json
{
  "hybridFts": {
    "profile": "zh-ngram",
    "dictVariant": null
  }
}
```

| `profile` | Lance `Index.fts` | 需词典 |
|-----------|-------------------|--------|
| **`zh-ngram`**（默认） | `ngram` 2～3 字；无 stemming / stop words | 否 |
| **`en`** | `simple` + English stemming / stop words | 否 |
| **`zh-jieba`** | `jieba/default` | 是（`dictVariant`：`small` / `default` / `big`） |

**复合规格键**（索引戳记、会话比对）：`formatHybridFtsSpec` → 如 `zh-ngram`、`zh-jieba:default`。各 Lance 目录旁写入 **`.hybrid-fts-profile.json`**；与当前全局设置不一致时，召回前会触发 FTS 重建。

**词典目录**（相对 `data/{userId}/`）：

```
hybrid-fts/
  zh-jieba/
    small/jieba/default/dict.txt
    default/jieba/default/dict.txt
    big/jieba/default/dict.txt
```

- 每个 profile+规格自带完整 **model 子树**；切换规格**只改**运行时 `LANCE_LANGUAGE_MODEL_HOME`，**不**再「复制到激活槽」。
- Lance jieba **不能**在 FTS 配置里直传词典绝对路径；运行时经 `withLanceLanguageModelHome` 将 `process.env.LANCE_LANGUAGE_MODEL_HOME` 设为上述规格的 **`hybrid-fts/{profile}/{variant}/`** 绝对路径（多用户并发时进程内 env 写入串行化，见 `lance-language-model-context.ts`）。
- **旧布局迁移**（读时一次性）：`lance-language-models/jieba/variants/{variant}/dict.txt` 或 `jieba/default/dict.txt` → 新树；迁移后旧目录可手动删除。

**会话戳记**：`chats/.../index.json` → **`memoryHybridFtsProfile`**（建索引时的 spec）。若全局 `hybridFts` 已变更而会话戳记未对齐，对话设置 UI 提示须重建该会话远期记忆索引。

**HTTP API**（均需 JWT）：

| 路由 | 说明 |
|------|------|
| `GET /api/hybrid-fts/catalog` | 各 profile 与 `zh-jieba` 规格目录（上游 URL、体积等） |
| `GET /api/hybrid-fts/dict-status?profile=` | 当前用户各规格是否已下载；**`modelHome` / `storagePath` 为相对 `data/{userId}/` 的路径** |
| `POST /api/hybrid-fts/dict-download?stream=1` | SSE 下载缺失词典到 `hybrid-fts/` |
| `PATCH` 用户偏好（既有路由） | body 含 `hybridFts: { profile?, dictVariant? }`；`zh-jieba` 切换前须词典就绪 |

**UI**：设置 → **记忆检索分词**（`SettingsView` + `HybridFtsSwitchDialog`）；对话侧 `ConversationContextSettings` 展示 FTS 规格不一致警告。

### 14.5 LanceDB、Syncthing 与损坏恢复

- **路径**：`{DATA_DIR}/{userId}/memory/conversations/{conversationId}/` → 表 `turn_memory`。
- **权威源**：`chats/` chunk JSON；Lance 为**可重建派生索引**。
- **单写者**（与 `DOC/01` §5.2 一致）：
  - 同一 `DATA_DIR` **只运行一个 server 进程**（勿 prod `npm start` 与 dev `npm run dev` 同时写盘）。
  - Syncthing：**同步 chunk**；**勿同步** `memory/` 半写入目录；各机本地重建索引更安全。
  - 删轮会 `scheduleMemoryIndexDelete`（Lance `delete by turnId`）；正常单进程下不会单独导致 fragment 丢失；**双进程写 / 同步冲突 / 写入中断**更易触发损坏。
- **损坏症状**：assemble / 预览报 Lance `Not found: ...turn_memory.lance/data/....lance` → API `memory_vector_index_corrupt`（503）；前端预览弹窗引导「重建远期记忆索引」。
- **恢复**：设置页重建（SSE `memory/rebuild?stream=1`）或删除该会话 `memory/conversations/{id}/` 后重建。检测：`server/src/memory-vector-index-error.ts`。

### 14.9 组装前统一管线（Memory / History / Lore）【部分已实现】

**核心约定**：**匹配**（用什么文本触发资料库、召回哪些 turn）与 **注入**（`messages[]` 里各槽顺序）**分离**。下列为匹配与计算顺序；注入顺序仍按 §14.3。

#### 14.9.1 阶段一：仅由 `userInput` 驱动 memory 与 history

```text
userText
  ├─► [Memory] embedding(buildMemoryRecallQuery)
  │         → LanceDB hybrid（FTS + vector + RRF，单次 TopK，depth=1，不递归）
  │         → turnId[] → resolveTurnById → memoryText（XML 内按 turnOrdinal 排序）
  │         （排除已在近 N 轮 history 窗口内的 turnId）
  │
  └─► [History] chunk 链 → 近 N 轮 turn → turnsToHistoryMessages / recentHistoryScanText
```

- **Memory 检索 query 不得**依赖尚未算出的 lore 正文（避免环）。
- **History** 不是向量「召回」，是固定滑动窗口。

#### 14.9.2 阶段二：资料库匹配（`userText + memory + history` → lore，递归 ≤2）

```text
scanCorpus = buildScanText(userText, memoryText, recentHistoryScanText)   // §13.5.1，可裁剪

matchedLore = resolveLoreRecursive(scanCorpus, maxDepth=2)      // §13.5.2
            → formatLoresXml*（单本扁平 / 多本分组）→ ctx.world
```

- 目的：用户本轮未打设定名，但 **召回旧轮或近期对白** 中出现关键字时，仍可触发资料库。
- 风险：误触发增多 → 用 `loreScanUserLinesOnly`、`maxScanChars`、条目去重与 `maxLoreTokens` 控制（§13.5）。

#### 14.9.3 阶段三：注入 `messages[]`（与匹配顺序不同）

```text
… character …
→ boundWorld（lore，阶段二结果）
→ boundMemory（memoryText，阶段一）
→ history 分组 / boundRecentHistory（ctx.history：user/assistant 链，阶段一）
→ post_history / 聊天中 …
→ boundUserInput（本轮 userText）
```

#### 14.9.4 跨模块禁止

| 禁止 | 原因 |
|------|------|
| lore 递归中调用 embedding / memory | 成本与语义漂移 |
| memory 命中后再跑第二轮向量检索 | 默认不做；若做须单独开关且限 1 次 |
| lore 命中触发 memory 再检索 | 易环；默认禁止 |
| 向量检索「多跳」（用召回 turn 再 embed 再搜） | 首版不做；见 §14.4；可选邻居扩展 1 hop |

#### 14.9.5 默认参数一览（会话级，规划）

| 模块 | 参数 | 默认 |
|------|------|------|
| History | `historyTurnCount` (N) | 12～20（实现时固化） |
| Memory | `memoryTopK` | 3～5 |
| Memory | 向量递归深度 | **1**（仅单次检索） |
| Lore | `loreMaxRecursionDepth` | **2** |
| Lore | `loreScanScope` | `userMemoryHistory` |
| Lore | `historyScanTurnCount` | ≤ N，如 6～10 |

### 14.10 Embedding 选型备忘（最低优先级 · 不排期）

> 实现讨论记录，供日后扩展检索时参考；**非当前交付范围**。

#### 向量维度

- **不是越大越好**：维度升高带来存储与检索成本；对本项目记忆 `memoryTopK`（约 3～5）、资料库小规模场景，**1024 维通常足够**。
- **MRL（Matryoshka）**：如 Qwen3-Embedding 标称支持 **32～1024** 多档输出；OpenAI 兼容 **`dimensions`** 需推理端真正支持（TEI、vLLM 等），否则网关可能忽略参数仍返回满维。
- **本项目**：`user-preferences` / 会话级 **`embeddingDimensions`**，留空则不传 `dimensions`；换支持 MRL 的服务即可生效，无需改 Lance 表结构（改维后须 **重建索引**）。
- **备选（未实现）**：客户端对满维向量截断前 N 维并 L2 归一化后写入 Lance（vLLM 文档所称 client-side MRL）。

#### Reranker（重排序）

- **已实现（hybrid）**：Memory 与资料库 vector 召回使用 Lance **`RRFReranker`** 融合 FTS（BM25）与向量 ANN 结果（见 §14.4.2）。**不**改变 keyword 匹配与 memory 预算裁切（按 `turnOrdinal` 删最早）。
- **未实现（cross-encoder 精排 · P3）**：① embedding + FTS hybrid **粗排** TopK；② **cross-encoder reranker** 对「query + 候选文档」联合打分 **精排**（准、成本高）。`api-settings.json` 已预留 **`rerank`** capability（见 §1.2）；`memoryTopK` 较小时收益有限，**不排期**。

#### Qwen Embedding 使用提示（未实现）

- 官方 README 建议：**query** 侧加任务指令前缀（如 `Instruct: Given a web search query, retrieve relevant passages\nQuery:`），**document** 侧不加；据称检索可提升约 **1～5%**。当前索引与检索均未拼接该前缀。

## 16. 正则替换（原生 · Phase 0–5 已落地 · 2026-06-10）

> 完整定案：**`DOC/24-regex-and-session-audit.md` §2**。本节为索引。

- **存储**：`data/{userId}/regex-rules.json`（**用户级**；无会话级字段）；规则含 `order`、`phases`、`fields`、`skipLastNTurns`。
- **执行**：enabled 规则按 `order` 升序在**内存**串联；**禁止**每条规则触发一次 chunk 写盘（历史批量走 `batchUpdateConversationTurns`）。
- **outgoing**：`assemble` → budget trim → **regex** → `afterAssemblePrompts` → upstream；**`<memory>` 块**内各轮 `<user>`/`<assistant>` 按 `turnOrdinal` 走与 history 相同的 `skipLastNTurns`（`regex-outgoing.ts` · `memoryItems`；见 **`DOC/24` §2.2**）。
- **persist**：`/api/chat` 落盘前；SSE persist 返回最终正文；UI 落盘后立刻更新。
- **display**：`use-regex-display-text` + 规则缓存。
- **宿主**：`host.regex` / server `api.regex`（`listRules` / `applyText` / `applyMessages`）。
- **UI**：**系统设置** `/settings` Tab 管理规则（拖曳、单条测试、管线测试）；**对话设置** →「正则批量」`ConversationRegexApplyPanel`（区间、规则只读勾选、dry-run / apply · `POST .../regex/apply`）。**未做**：export 勾选（可选）。
- **新建规则默认**：`phases: ['display']`  only（避免误勾 outgoing/persist）；种子「规范省略号」同为 `display` only、默认 `enabled: false`。
- **replacement**：`String.replace` 原生语义；**真实换行**有效，字面量 `\n` 两字符**不**转义；`$1` / `$&` / `$$` 等按 JavaScript 替换语法；设置页为 **textarea**。
